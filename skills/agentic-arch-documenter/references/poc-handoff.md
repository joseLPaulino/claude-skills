# POC Handoff

Use this when the user wants to move from Level 1 architecture into a proof of concept.

## POC Scope

Keep the first POC narrow:

- one primary use case
- one or two representative user journeys
- limited data sources
- simulated or read-only integrations where production access is risky
- visible trace, audit, and approval behavior
- measurable success criteria

## POC Outputs

Recommend outputs such as:

- demo UI or workflow
- API/backend service
- agent orchestration trace
- sample decisions
- evaluation results
- architecture update
- backlog for production hardening

## POC Readiness Checklist

- source data identified
- test cases defined
- platform access confirmed
- model/provider constraints confirmed
- security review path identified
- demo story agreed
- success metrics agreed
- known non-production shortcuts documented

## Do Not Overbuild

Avoid production hardening in the first POC unless it is the point of the POC. Clearly separate demo scaffolding from production architecture.
